# Plan: Integración con Base de Datos Externa (an_core)

## Contexto

### Proyecto Actual (uyuni-backend-py)
- **Base de datos**: `fastapi_product` (PostgreSQL)
- **ID Type**: UUID v7 (`uuid6.uuid7`) - Primario
- **External ID Pattern**: YA EXISTE - Campos `external_id: int` en Staff, OrgUnit, StaffPosition
- **ORM**: SQLModel (async)

### Proyecto Externo (get-an-personal-data)
- **Base de datos**: `an_core` (PostgreSQL)
- **ID Type**: Integer Sequential (auto-increment)
- **Tablas principales**:
  - `funcionario` → Staff (personal/empleados)
  - `cargo` → StaffPosition (cargos)
  - `unidad_organizacional` → OrgUnit (unidades organizacionales)
- **Fuentes**: SIGER API, SAU, SAOMCO, Nande, Suite, Viajero
- **ORM**: SQLModel

### Mapeo de Tablas

| an_core (Integer ID) | fastapi_product (UUID + external_id) |
|----------------------|--------------------------------------|
| `funcionario.id` | `core_staff.id` (UUID) + `external_id` (int) |
| `cargo.id` | `core_staff_position.id` (UUID) + `external_id` (int) |
| `unidad_organizacional.id` | `core_org_unit.id` (UUID) + `external_id` (int) |

## Opciones de Integración (Estándar de Industria)

### Opción A: Database Link (PostgreSQL Foreign Data Wrapper) ⭐ RECOMENDADO

**Concepto**: Usar FDW de PostgreSQL para acceder a `an_core` como si fuera local.

```
┌─────────────────────────────────────────────────────────────┐
│                    fastapi_product                          │
│  ┌─────────────┐     ┌──────────────────────────────────┐  │
│  │ core_staff  │     │      Foreign Table (FDW)         │  │
│  │ (local)     │────▶│  funcionario (remote via FDW)    │  │
│  │ UUID + ext  │     │  READ-ONLY access                │  │
│  └─────────────┘     └──────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼ FDW Connection
                    ┌─────────────────┐
                    │    an_core      │
                    │  (funcionario)  │
                    └─────────────────┘
```

**Ventajas**:
- Sin duplicación de datos
- Consistencia garantizada (lectura en tiempo real)
- Transparencia total para la aplicación
- Query SQL joins entre bases de datos
- PostgreSQL nativo (sin dependencias)

**Desventajas**:
- Latencia de red en queries
- `an_core` debe estar siempre disponible
- Sin cache local

**Implementación**:
```sql
-- En fastapi_product
CREATE EXTENSION postgres_fdw;
CREATE SERVER an_core_server FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host 'localhost', dbname 'an_core', port '5432');
CREATE USER MAPPING FOR postgres SERVER an_core_server
  OPTIONS (user 'postgres', password 'postgres');
  
CREATE SCHEMA external;
IMPORT FOREIGN SCHEMA public FROM SERVER an_core_server INTO external;
-- Ahora: SELECT * FROM external.funcionario;
```

---

### Opción B: ETL/Sincronización Periódica (Data Replication)

**Concepto**: Job programado que replica datos de `an_core` a `fastapi_product`.

```
┌─────────────┐         Sync Job          ┌─────────────────┐
│  an_core    │ ───────────────────────▶  │ fastapi_product │
│ (SOURCE)    │    (cron/celery/airflow)  │   (REPLICA)     │
└─────────────┘                           └─────────────────┘
```

**Ventajas**:
- Datos locales (sin latencia)
- `an_core` puede estar offline
- Cache implícito
- Control total de los datos

**Desventajas**:
- Datos potencialmente desactualizados
- Complejidad de sincronización
- Duplicación de storage
- Conflictos de escritura

**Implementación**:
- Script Python con SQLModel
- Upsert por `external_id`
- Soft delete para registros removidos
- Scheduler: Celery, APScheduler, o cron

---

### Opción C: API Gateway / BFF (Backend for Frontend)

**Concepto**: Endpoint que consulta ambas bases y combina resultados.

```
┌────────────────────────────────────────────────────────────┐
│                    FastAPI Endpoint                        │
│                                                            │
│   GET /staff/external/{id}                                 │
│   ┌─────────────┐              ┌─────────────────────┐    │
│   │ Session 1   │              │     Session 2       │    │
│   │ (local DB)  │              │  (external DB)      │    │
│   └─────────────┘              └─────────────────────┘    │
│         │                              │                  │
│         ▼                              ▼                  │
│   fastapi_product                an_core                  │
│   (core_staff)                   (funcionario)            │
└────────────────────────────────────────────────────────────┘
```

**Ventajas**:
- Sin configuración de DB
- Flexibilidad total en queries
- Fácil de implementar

**Desventajas**:
- Latencia duplicada
- Código más complejo
- Sin SQL joins nativos

**Implementación**:
```python
# app/core/external_db.py
from sqlalchemy import create_engine
from app.core.config import settings

EXTERNAL_DATABASE_URL = "postgresql://postgres:postgres@localhost/an_core"
external_engine = create_engine(EXTERNAL_DATABASE_URL)

def get_external_session():
    with Session(external_engine) as session:
        yield session
```

### Opción D: Event-Driven (CDC - Change Data Capture)

**Concepto**: Escuchar cambios en `an_core` y propagar eventos.

```
┌─────────────┐     WAL/Trigger     ┌─────────────┐     Event     ┌─────────────────┐
│  an_core    │ ─────────────────▶  │   Kafka/    │ ────────────▶ │ fastapi_product │
│             │                     │   RabbitMQ  │               │                 │
└─────────────┘                     └─────────────┘               └─────────────────┘
```

**Ventajas**:
- Tiempo real
- Desacoplamiento total
- Escalable

**Desventajas**:
- Infraestructura compleja (Kafka, Debezium)
- Overkill para este caso

**Respuestas confirmadas**:
- ⏱️ **Delay tolerable**: Sí, datos pueden tener minutos/horas de desfase
- ✅ **Disponibilidad**: `an_core` siempre disponible
- 📊 **Volumen**: < 10,000 registros (bajo volumen)

## Solución Recomendada: Opción B (ETL/Sync Periódico) ⭐

**Por qué es la mejor opción para tu caso**:
1. ✅ **Tolerancia a delay**: No necesitas tiempo real
2. ✅ **Volumen bajo**: < 10K registros = sync rápido
3. ✅ **Ya tienes `external_id`**: Infraestructura lista
4. ✅ **Autonomía**: Datos locales, sin dependencia de `an_core`
5. ✅ **Performance**: Sin latencia de red en queries

class SyncService:
    def __init__(self, local_session: Session, external_session: Session):
        self.local = local_session
        self.external = external_session
    
    async def sync_org_units(self) -> SyncResult:
        """Sync unidad_organizacional → core_org_unit"""
        external_units = self.external.exec(select(ExternalUnidadOrganizacional)).all()
        
        synced = 0
        for ext in external_units:
            # Buscar por external_id
            local = self.local.exec(
                select(OrgUnit).where(OrgUnit.external_id == ext.id)
            ).first()
            
            # Mapear datos
            data = {
                "external_id": ext.id,
                "name": ext.nombre,
                "acronym": ext.sigla,
                "general_unit": ext.unidad_general,
                "type": "MANAGEMENT" if ext.tipo == "GERENCIA" else "DEPARTMENT",
            }
            
            if local:
                # Update
                for key, value in data.items():
                    setattr(local, key, value)
            else:
                # Create
                local = OrgUnit(**data)
                self.local.add(local)
            synced += 1
        
        self.local.commit()
        return SyncResult(entity="org_units", synced=synced)

### Fase 6: Mapeo de Relaciones

**Desafío**: Las FKs en `an_core` son integers, en `fastapi_product` son UUIDs.

**Solución**: Resolución en tiempo de sync:
```python
# En sync_staff:
ext_funcionario = external_session.get(ExternalFuncionario, ext_id)

# Resolver FKs de integer → UUID
position = local_session.exec(
    select(StaffPosition).where(StaffPosition.external_id == ext_funcionario.cargo_id)
).first()

org_unit = local_session.exec(
    select(OrgUnit).where(OrgUnit.external_id == ext_funcionario.gerencia_id)
).first()

# Crear/actualizar con UUIDs resueltos
staff_data = {
    "external_id": ext_funcionario.id,
    "position_id": position.id if position else None,
    "org_unit_id": org_unit.id if org_unit else None,
    # ... otros campos
}
```
